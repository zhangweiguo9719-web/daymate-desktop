# 技术架构

## 目标

DayMate 的核心不是“采集越多越好”，而是在最小采集范围内回答三个问题：昨天主要做了什么、今天先做什么、怎样更容易开始。

## 分层

1. `src/App.tsx`：页面组合与用户交互，不直接访问 Windows API。
2. `src/store.ts`：任务、偏好和本地规则；`selectNextTask` 可独立测试。
3. `src/native.ts`：前端唯一桌面桥接层；浏览器预览会安全回退为空数据。
4. `src-tauri/src/lib.rs`：系统采集、SQLite、Tauri 命令和托盘。

## 活动数据流

```text
每 3 秒采样前台窗口
  → 获取进程名
  → 可选获取窗口标题
  → 检查空闲是否达到 5 分钟
  → 应用切换 / 跨天 / 60 秒时刷新会话
  → SQLite WAL 批量写入
  → get_today_stats 聚合查询
  → React 时间回顾页
```

DayMate 自身不计入有效应用时长。窗口标题只有用户主动开启后才进入内存和数据库。

## 数据表

- `migration_history`：数据库结构版本和应用时间。
- `app_usage_sessions`：应用名、可选标题、开始/结束、活跃/空闲秒数。

查询按开始时间和应用名称建索引。数据库采用 WAL，采样不会每三秒直接写盘。

## 本地规则

“帮我选一件”依次比较：截止日期、优先级、预计用时。已完成任务永不参与选择；没有 AI Key 时功能完整可用。

## 后续演进

- 把任务和偏好从 WebView 本地存储迁移到 SQLite，并提供一次性迁移。
- 增加按日/周查询和应用分类表。
- 接入 Tauri 官方 autostart/notification 插件。
- AI 保持可选，只接收汇总数据，失败自动回退到本地规则。
